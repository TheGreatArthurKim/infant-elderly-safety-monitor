# 비접촉 입욕 안전 모니터링 시스템

욕조에 넣는 완전 방수 압력 센서로 **정상 입욕과 위험 상황(익수 가능성)을 구분**해 보호자에게 알리는 비접촉 안전 디바이스. 카메라·웨어러블을 쓰지 않는다. 영유아 욕조 익사와 노약자 입욕 사고를 1차 대상으로 한다.

핵심 IP는 **수중 압력의 시간 패턴(dP)으로 정상 입욕과 익수를 분리**하는 것이다.

> **현재 단계 = Phase A.** "압력 데이터만으로 정상 입욕과 익수가 실제로 분리되는가"를 검증 중이다. 이 검증(GO/KILL)이 통과되기 전에는 Sub-GHz 통신·메인 허브·클라우드·앱·양산 하드웨어를 착수하지 않는다.

> 기술적 한계·미검증 항목은 [RISKS.md](./RISKS.md)에 사전 공개한다. 모든 정확도·분해능 수치는 실측 대체 전까지 데이터시트/참고문헌 **인용 기준**이다.

---

## 핵심 설계 원칙

| 원칙 | 내용 |
|---|---|
| 1차 신호 = 수중 압력 | 잠긴 신체 부피 변화 → 수면 변화 → 압력. 카메라·레이더가 못 보는 수중을 물리적으로 직접 측정 |
| 스칼라 dP로는 부족 | 격한 물장구와 익수는 **크기가 겹친다.** 실제 분리축은 **시간 패턴**(복귀형 vs 지속형) |
| 생명 알림은 로컬 우선 | 인터넷이 끊겨도 동작해야 한다. 클라우드·앱은 원격 조회 보조, **응급 경보는 로컬(허브 자체)** |
| 가역성 | 검증 단계 — 단일 센서. 비가역 하드웨어 처리(실리콘 밀봉 등) 금지 |

## 상용 제품 구조 (V1 목표)

```
방수 센서 모듈(ESP32 + MS5803 + 배터리 + Sub-GHz)
        │  Sub-GHz
        ▼
메인 허브(게이트웨이) ──[로컬 우선 경보: 사이렌·LED·LAN 푸시]
        │  인터넷
        ▼
클라우드 ──▶ 스마트폰 앱(원격 조회·이력)
```

허브는 **로컬 우선 경보 + 개발자 모드(PC 실시간·CSV·디버깅)**를 함께 지원한다. **단, 이 구조 전체는 Phase A 통과 후 착수한다.** 현재는 개발 편의를 위해 Wi-Fi/TCP/`logger.py`를 그대로 사용한다.

## 개발 로드맵 · 완료 조건(Exit Criteria)

기술 리스크를 먼저 제거하는 순서로 단계화한다. 각 단계는 완료 조건을 넘겨야 다음으로 간다.

| 단계 | 목표 | 완료 조건(Exit Criteria) |
|---|---|---|
| **트랙0 (병렬)** | 규제·IP | 식약처 의료기기 해당여부 **사전질의 1차 답변** 확보 + 특허 선행조사 후 **출원/유예 결정**(공개 전 출원 원칙) |
| **A. 압력 검증** | 구조화 데이터 수집 + 통계 분석 → 판정 | 아래 [Phase A 판정 기준](#phase-a-판정-기준-확정) 충족 + 한계(Risk 02·11·12) 명문화 |
| ** 진행 / 피벗** | 진행 또는 피벗 결정 | 진행 = 분리 성공 / 피벗 = 실패 → 역할 재정의(입수·퇴수 검출 + 비정상 패턴) + 2차 모달리티 추가 |
| **B. 알고리즘 강건화** | 온디바이스 실시간 분류 | **미학습 욕조에서 재현** + 오탐률 목표 + 실시간 판정 지연 목표 |
| **C. Sub-GHz 링크** | Wi-Fi → Sub-GHz 교체 | 욕실 벽 통과 **PER < 1%** + 알림 지연 < 1s + 배터리 목표 |
| **D. 허브 게이트웨이** | 로컬 우선 알림 + 수신/ACK/상태 | **인터넷이 끊겨도 로컬 경보 동작** + 패킷 신뢰성(ACK/Retry) |
| **E. Cloud + 앱** | 원격 조회 + 푸시 | 외부 망에서 상태 조회 + 푸시 도달률 목표 |
| **F. PCB · 인증** | 양산 HW + 방수 신뢰성 + 인증 | 장기 결로·방수 신뢰성 통과 + 인증 통과 |

지금 시작하는 것은 **트랙0과 Phase A뿐.** C~F는 게이트 통과 후 전개한다.

### Phase A 판정 기준 (확정)

충돌쌍(정상 vs 익수가 가장 헷갈리는 쌍: N4-D5, N2-D3, N5-D2)에서 판정한다.

| 게이트 | 지표 | 기준 | 비고 |
|---|---|---|---|
| 신호 존재 | SNR | **≥ 10 dB** | 정의 `20·log10(\|peak dP\| / σ)`, σ=0.5mbar → **peak dP ≥ 1.58mbar** |
| 분리력 (핵심 IP) | AUC | **≥ 0.90** | 충돌쌍별, 특징 F1~F6 통합 분류기, Leave-One-Trial-Out CV |
| 진단 (게이트 아님) | Cohen's d | 핵심 특징 **≥ 1.5** | 어떤 특징이 신호를 싣는지 + 과적합 감별 |
| KILL / 피벗 | AUC | 어느 충돌쌍이라도 **< 0.75** | 압력 단독 분리 실패 → 피벗 |

> 두 정규분포 가정에서 `AUC = Φ(d/√2)`이므로 d와 AUC는 서로 다른 축(단일특징 vs 통합분류기)으로 분리해 사용한다. 원안의 `d≥0.8 AND AUC≥0.9`는 부정합(AUC≥0.9 ⇒ d≈1.81)이라 폐기했다.

## 저장소 구조

```
firmware/
  sensor-node/
    pressure.ino          # ESP32 — 압력·온도·baseline·dP, WiFi/TCP/LittleFS 로거 (Phase A)
tools/
  logger.py               # PC — TCP 수신 → CSV 저장, 덤프(r)/삭제(c)/재개(s), SYNC 시각 동기
RISKS.md                  # 기술적 한계·검증 계획
system-design-document.*  # 상세 설계서 (압력 제품 + 장기 융합 비전)
```

| 영역 | 상태 |
|---|---|
| 압력 센서 단독 로거 (Phase A) | 구현 완료 — `sensor-node/pressure.ino` + `tools/logger.py` |
| 측정 품질 업그레이드 | 예정 — 샘플링 2→10Hz(압력)/1Hz(온도), 정밀 타임스탬프, 시나리오·라벨 CSV 스키마 |
| 데이터 수집 + 통계 분석 (Phase A 본체) | 진행 예정 — 특징 F1~F6, 충돌쌍 매트릭스 |
| Sub-GHz·허브·클라우드·앱·PCB | 미착수 (Phase A 통과 후) |

## 장기 비전 — 멀티센서 융합 (현재 범위 밖)

거실·안방 메인 허브 + 욕실 워치독으로 **낙상·호흡·체온·활동 패턴까지 통합**하는 융합 안전 시스템. 60GHz mmWave + IR Thermal + PIR 융합, 조명 일체형. 상세 설계는 `system-design-document`에 있으며, **Phase A 압력 검증이 통과된 이후 단계적으로 추진**한다. 이 비전의 가설·한계는 RISKS.md에 `[장기 비전]` 태그로 별도 표기한다.

## 차별화

1. **영유아 욕조 비접촉 익수 감지** — 확인된 범위 내 글로벌 상용 사례 없음 (핵심)
2. **로컬 우선 안전 구조** — 인터넷 의존 없이 응급 경보 보장
3. **(비전) 조명 일체형 + 자녀↔부모 양방향 연결** — 설치 거부감 제거, 단방향 알림 경쟁사 대비 차별
